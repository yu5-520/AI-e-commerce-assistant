# 产品经理视角复盘

## 1. 项目解决的问题

真实电商团队不是只缺一条标题或一段主图文案，而是缺少把商品、订单、库存、客户、售后和复盘串起来的自动化工作流。

本项目尝试解决：

- 经营数据分散
- 商品测试不可追溯
- 客户分层不清晰
- 售后问题无法反向影响商品运营
- RPA 自动化缺少风险边界
- AI 输出缺少业务依据和评测机制

## 2. 产品设计选择

### 2.1 选择 Workflow-first

原因：

- 电商场景风险高。
- ERP / CRM 数据涉及真实经营和隐私。
- 高风险动作需要人工确认。
- 固定流程更容易验收和追溯。

### 2.2 暂不选择 Agent-first

原因：

- 不希望 AI 自主改价、投放、触达客户。
- 不希望系统绕过平台规则和权限边界。
- 当前阶段更适合先验证业务闭环。

### 2.3 先用 Mock 数据

原因：

- 避免真实客户隐私风险。
- 避免平台账号和接口权限问题。
- 更适合 MVP 试运行。

## 3. 产品闭环

```text
Mock ERP / CRM 数据
↓
AI / 规则诊断
↓
RAG 召回依据
↓
RPA 任务草案
↓
人工确认
↓
报告输出
↓
下一轮复盘
```

## 4. 当前价值

当前项目已经证明：

- 可以把 AI 放进业务流程，而不是只做内容生成。
- 可以把 ERP / CRM 数据纳入 AI 工作流。
- 可以通过 Human-in-the-loop 控制高风险动作。
- 可以用 Evals 验证自动化边界。
- 可以用前端 Demo 对外展示流程。

## 5. 当前不足

- 未接真实模型 API。
- 未接真实 ERP / CRM。
- RAG 仍是关键词检索。
- 前端还未连接后端 API。
- 任务确认状态未落数据库。

## 6. 产品下一步

最优先：

```text
FastAPI 封装 Python Workflow
↓
前端 fetch 调用 API
↓
用户确认 / 拒绝任务
↓
日志回写
```

## 7. 复盘结论

这个项目的产品价值不在于“大而全”，而在于：

> 用一个可运行 MVP 证明 AI 工作流可以安全接入电商经营流程。
