# MVP PRD：AI 垂直货架电商经营循环系统

## 1. MVP 目标

第一版不做完整电商 ERP，不直接接入真实店铺后台，也不做无约束 Agent 自治。

当前 V0.8 只验证一个核心价值：

> 商家能否基于 Mock ERP / CRM 数据，完成“数据导入 → 经营判断 → RPA 任务草案 → 人工确认 → 报告与日志回写”的最小经营闭环。

后续版本在这个经营判断底座上继续扩展：

```text
垂直类目配置
↓
同类目竞品比对
↓
同类目上新增长
↓
流量测试与数据回流
↓
回到经营判断
```

## 2. 产品定位

产品名称：AI 垂直货架电商经营循环系统 MVP

一句话介绍：

> 面向垂直类目货架电商商家的 Workflow-first 系统：先接入已有商品、库存、订单、退款、客户和售后数据完成经营判断；再由经营问题触发同类目竞品比对；再生成优化、上新和流量测试方案；最后把测试结果回流到经营判断系统。

当前版本关键词：

- 垂直类目经营
- Workflow-first
- Agent-ready
- ERP / CRM Mock 数据
- 规则引擎模拟 AI 诊断
- 轻量 RAG 知识增强
- RPA 任务草案
- Human-in-the-loop
- SQLite / JSONL 日志回写

## 3. MVP 用户

### 3.1 核心用户

- 垂直类目货架电商小商家
- 白牌商品商家
- 有固定货源和垂直类目经验的店主
- 电商运营新人
- 有 ERP / 表格数据但不会系统分析的个体商家
- 想用 AI 辅助经营但不希望直接自动操作店铺后台的商家

### 3.2 用户痛点

- 商品、订单、库存、退款、客户数据分散在不同表格里。
- 老板懂货，但每天看表、写日报、做复盘很耗时。
- 不知道哪些 SKU 有利润风险。
- 不知道库存高但订单低时该先清货、测款还是降价。
- 不知道退款率异常应该先查商品、详情页、物流还是客服。
- 不知道客户应该如何分层、复购、召回或先处理售后。
- 不知道什么时候该看竞品，什么时候该扩相似品或上新。
- AI 工具输出很多，但没有审批、日志和复盘链路。
- RPA 工具能执行动作，但缺少风险边界和人工确认。

## 4. MVP 不做什么

第一版明确不做：

- 不接真实店铺后台。
- 不自动上架。
- 不自动改价。
- 不自动报名活动。
- 不自动投放广告。
- 不自动群发客户消息。
- 不自动退款。
- 不绕过验证码或平台风控。
- 不提供违规爬虫方案。
- 不保存真实姓名、手机号、微信号、地址等客户隐私。
- 不做复杂多租户权限。
- 不做完整企业级 ERP / CRM。
- 不宣称已经完成真实竞品抓取、自动上新或自动投流。

第一版只做：

> Mock 数据验证 + 经营判断 + 任务草案 + 人工确认 + 报告日志。

## 5. 产品阶段边界

### V0.8：已有商品经营判断底座（当前）

目标：先帮商家看懂已有生意。

包含：

- 数据导入与校验。
- 商品经营诊断。
- CRM 客户分层。
- RAG 知识召回。
- RPA 任务草案。
- 人工审批。
- 报告与日志。

### V0.9：垂直类目配置层

目标：让系统从泛电商判断升级为垂直类目判断。

包含：

- 类目知识档案。
- 价格带配置。
- 季节性配置。
- 高频售后问题。
- 主图表达模板。
- SKU 结构模板。
- 竞品比对维度。

### V1.0：同类目竞品比对系统

目标：经营判断发现问题后，能找同类目竞品做差距分析。

包含：

- 竞品 Mock 数据。
- 价格带比对。
- 标题关键词比对。
- 主图卖点比对。
- SKU 结构比对。
- 差评机会分析。
- 是否优化老品 / 扩相似品 / 暂停投入。

### V1.1：同类目上新增长系统

目标：从已有经营数据、竞品差距和供应链货盘里生成上新方案。

包含：

- 货盘导入。
- 新品候选评分。
- 标题草案。
- 主图文案草案。
- SKU 草案。
- 定价草案。
- 上架字段草案。
- 合规检查表。
- 首轮流量测试计划。

### V1.2：流量测试与数据回流系统

目标：让新品和优化方案进入可复盘的测试循环。

包含：

- 流量测试计划。
- 曝光、点击、转化、退款、ROI 回流。
- 点击问题诊断。
- 转化问题诊断。
- 价格 / SKU / 主图 / 售后问题判断。
- 下一步动作决策。
- 回写经营判断系统。

## 6. V0.8 核心功能

## 6.1 数据导入与校验

### 功能说明

用户通过 Mock CSV / Excel 样例导入 ERP / CRM 数据，系统检查字段、空值、数字格式和表关系。

### 输入数据

- 商品表
- 订单表
- 库存表
- 退款表
- 客户表
- 客户标签表
- 客户互动表

### 输出

- 数据源列表
- 字段完整性校验
- 数值合法性校验
- 商品 / 订单 / 库存 / 退款关系校验
- 客户 / 标签 / 互动关系校验
- ImportRecord
- WorkflowRun / ExecutionLog

### 验收标准

- 能列出所有 Mock 数据源。
- 能识别缺失字段、空必填项、非法数字。
- 能检查订单、库存、退款是否关联到存在的商品。
- 能检查客户标签、互动是否关联到存在的客户。
- 点击导入后能生成导入记录。

## 6.2 商品经营诊断

### 功能说明

基于商品、订单、库存、退款数据，判断经营风险并输出建议动作。

### 输入

- 成本价
- 售价
- 活动价
- 物流成本
- 库存
- 订单数量
- 实付金额
- 退款记录
- 是否敏感类目

### 诊断内容

- 售价是否低于成本。
- 活动价是否接近或低于保本线。
- 是否高库存低订单。
- 退款率是否异常。
- 是否需要合规检查。
- 是否应触发竞品比对。
- 是否存在上新 / 扩品机会。

### 输出

- 商品 ID
- 商品名称
- 订单数
- 销售额
- 库存
- 退款数
- 退款率
- 毛利
- 活动价毛利
- 风险等级
- 风险原因
- 建议动作
- trigger_next_step

### 验收标准

- 活动价有亏损风险时必须提示人工确认。
- 高库存低订单时必须提示先做测试或清货测算。
- 退款率异常时必须进入售后归因方向。
- 敏感类目必须提示合规检查。
- 低点击 / 低转化 / 高退款 / 高库存问题后续可触发竞品比对。

## 6.3 CRM 客户分层

### 功能说明

基于客户订单、金额、退款、RFM、标签和互动记录，对客户进行经营分层。

### 输入

- 客户 ID
- 昵称哈希
- 总订单数
- 总消费金额
- 退款次数
- RFM 分数
- 客户标签
- 互动记录
- 情绪标记

### 分层类型

- 高价值客户
- 新客
- 沉睡客户
- 售后敏感客户
- 普通客户

### 输出

- 客户 ID
- 分层结果
- RFM
- 标签
- 判断依据
- 推荐动作
- 风险等级
- 是否需要人工确认
- 是否允许自动执行

### 验收标准

- 高价值客户可生成复购任务草案。
- 沉睡客户可生成低频召回任务表。
- 售后敏感客户必须先进入售后归因，不直接营销触达。
- 所有客户触达相关任务默认不自动执行。

## 6.4 RAG 知识增强

### 功能说明

系统从知识库中召回平台规则、合规风控、运营方法和客服 SOP，为诊断与任务草案提供依据。

### 当前版本

- 使用关键词检索模拟 RAG。
- 从 `knowledge_base/*.md` 读取知识片段。
- 返回 source、score、snippet。

### 后续版本

- 增加 category profile。
- 增加 metadata。
- 增加 retrieval trace。
- 接入 Embedding。
- 接入向量库。
- 将 RAG 证据传入 LLM 诊断节点。

### 验收标准

- 活动价风险能召回活动价 / 保本线相关知识。
- 售后风险能召回客服 SOP / 售后归因相关知识。
- 客户触达风险能召回隐私与群发边界相关知识。
- 垂直类目问题能召回对应类目知识。

## 6.5 RPA 任务草案

### 功能说明

系统根据诊断结果生成任务草案，但不执行真实平台动作。

### 任务类型

- 经营日报
- SKU 价格建议表
- 活动准备表
- 客户分层报告
- 复购任务表
- 售后归因表
- 复盘报告

后续扩展：

- 竞品比对报告
- 上新检查表
- 流量测试表
- 周度复盘报告

### 风险分级

- 高风险：自动改价、自动报名活动、自动加预算、自动群发、自动退款、自动上下架。
- 中风险：活动准备、客户话术、SKU 价格表、召回任务表、上新字段草案。
- 低风险：日报、分层报告、售后分析、复盘报告、竞品分析报告。

### 验收标准

- MVP 阶段所有任务都需要人工确认。
- 所有任务 `auto_execution_allowed` 必须为 false。
- 高风险任务只能输出建议，不进入自动执行。
- 任务必须包含 policy_reason。

## 6.6 人工审批中心

### 功能说明

用户可以对任务草案进行确认或拒绝，系统记录审批状态。

### 输出

- task_id
- approval_id
- workflow_run_id
- approval_status
- operator
- updated_at
- execution_note

### 验收标准

- approve / reject 后状态可查询。
- 审批记录写入 SQLite。
- 审批记录写入 JSONL。
- 审批动作不触发真实 RPA 执行。

## 6.7 报告与日志中心

### 功能说明

系统生成报告，并记录每次工作流的运行节点。

### 日志对象

- WorkflowRun
- ExecutionLog
- ImportRecord
- ApprovalRecord
- TaskStatus
- ReportRecord

### 验收标准

- 能查看最近工作流运行。
- 能查看节点级日志。
- 能按 workflow_run_id 查询节点。
- 能查看报告记录。
- 能追踪某个任务从生成到审批的状态变化。

## 8. MVP 成功指标

### 8.1 使用指标

- 用户能否完成 Mock 数据导入。
- 用户能否读懂商品诊断结论。
- 用户能否理解客户分层结果。
- 用户能否确认或拒绝任务草案。
- 用户能否查看报告和日志。

### 8.2 产品价值指标

- 是否减少人工整理经营数据的时间。
- 是否帮助用户识别 SKU / 库存 / 售后风险。
- 是否帮助用户把客户运营从“群发”变成“分层处理”。
- 是否让 AI 输出具备审批、日志和复盘闭环。
- 是否能从经营问题自然引出竞品比对和上新增长。

### 8.3 安全指标

- 不出现自动改价。
- 不出现自动投放。
- 不出现自动群发。
- 不出现自动退款。
- 不出现绕过平台风控建议。
- 高风险任务必须进入人工确认。

## 9. 结论

当前 MVP 不追求一口吃成“全自动货架电商”，而是先建立最重要的底座：

```text
已有商品经营判断
↓
任务草案
↓
人工确认
↓
报告与日志
```

后续再从这个底座继续长出：

```text
垂直类目配置
↓
同类目竞品比对
↓
同类目上新增长
↓
流量测试回流
↓
经营判断循环
```

只要这个循环成立，产品就从普通 AI 工作流升级成垂直类目货架电商的可控自动运营系统。
