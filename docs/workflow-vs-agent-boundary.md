# Workflow 与 Agent 边界设计

## 1. 设计结论

本项目当前不是深度 Agent 自治系统，而是 **Workflow-first 的多系统连接型 AI 工作流**。

也就是说，系统的核心不是让 AI 自主运营店铺，而是将 ERP、CRM、RAG、AI、人工确认、RPA 和日志回写串成一条可控业务流程。

## 2. 为什么不是深度 Agent

深度 Agent 的典型特征是：

```text
目标理解
↓
自主拆解任务
↓
自主选择工具
↓
执行动作
↓
观察结果
↓
重新规划
↓
循环直到完成目标
```

而本项目当前流程是：

```text
预设业务流程
↓
AI 在指定节点生成判断
↓
人工确认关键动作
↓
RPA 执行低风险任务
↓
日志回写
↓
复盘迭代
```

AI 是流程中的判断节点，不是无边界的自主大脑。

## 3. 为什么采用 Workflow-first

电商 ERP / CRM / RPA 场景涉及真实经营风险：

- 改价会影响利润。
- 活动报名会影响库存和毛利。
- 投放会消耗真实预算。
- 客户触达涉及隐私与骚扰风险。
- 售后处理涉及用户权益和平台规则。

因此，企业级 AI 应用更适合：

- 固定流程
- 明确权限
- 人工确认
- 可回滚执行
- 日志追溯
- 风险审计

## 4. 当前系统边界

当前 AI 可以做：

- 商品诊断
- SKU 利润测算
- 库存预警
- 客户分层
- 复购判断
- 售后归因
- 活动准备建议
- RPA 任务草案
- 复盘报告

当前 RPA 可以做：

- 生成日报
- 导出表格
- 整理测试记录
- 生成客户清单
- 生成活动准备表
- 生成售后归因表
- 回写本地日志

当前系统不允许自动执行：

- 自动改价
- 自动上架
- 自动报名活动
- 自动投放广告
- 自动下架商品
- 自动群发客户
- 自动处理退款
- 自动诱导好评

## 5. Agent-ready 位置

未来可在低风险环节引入 Agent：

### 5.1 数据异常排查 Agent

自动检查：

- 哪个字段缺失
- 哪个表格式异常
- 哪个 product_id 无法匹配
- 哪个 customer_id 缺少订单记录

### 5.2 RAG 检索选择 Agent

根据任务自动选择检索：

- 商品档案库
- 客户档案库
- 平台规则库
- 合规风控库
- 运营方法库
- 客服 SOP 库

### 5.3 报告生成 Agent

根据日报、库存、订单、退款和客户分层生成复盘报告。

### 5.4 RPA 失败原因分析 Agent

当 RPA 执行失败时，分析失败原因并给出人工处理建议。

### 5.5 活动准备任务拆解 Agent

将“准备一场活动”拆解为：

- 库存检查
- 价格检查
- 毛利检查
- 主图检查
- 标题检查
- 合规检查
- 客户触达草案

## 6. 高风险 Agent 禁区

以下动作不交给 Agent 自主执行：

- 改价
- 投放
- 报名活动
- 自动触达客户
- 自动退款
- 自动下架
- 自动修改主图
- 自动修改 ERP / CRM 关键字段

## 7. 简历表达建议

建议写：

> 采用 Workflow-first 架构，将 ERP / CRM 数据接入、RAG 知识增强、AI 诊断、Human-in-the-loop 人工确认、RPA 低风险执行和日志回写串联为可控业务闭环；当前不追求 Agent 深度自治，而是在低风险环节预留 Agent-ready 扩展点。

不建议写：

> 已实现 AI Agent 自主运营系统。

## 8. 设计结论

本项目的核心价值是：

> 不是让 AI 自主接管电商运营，而是把 AI 安全地嵌入真实业务流程，让每个判断、确认、执行和回写动作都可控、可追溯、可复盘。
