# Human-in-the-loop 人工确认机制设计

## 1. 设计目标

本项目采用 Human-in-the-loop 机制，将 AI 与 RPA 的能力限制在可控范围内。

核心原则：

> AI 可以判断，RPA 可以整理和执行低风险任务，但凡涉及资金、商品状态、平台规则和不可回滚操作，都必须由用户人工确认。

## 2. 为什么需要人工确认

电商运营涉及真实经营风险：

- 改价可能造成亏损。
- 报名活动可能造成库存和利润压力。
- 投放广告涉及真实资金消耗。
- 修改标题 / 主图可能影响流量和转化。
- 下架或清货会影响商品生命周期。
- 违规表达可能触发平台审核和处罚。

因此，AI 与 RPA 不能绕过用户直接执行关键动作。

## 3. 动作风险分级

### 3.1 低风险动作

可由 RPA 在确认后执行，部分场景可默认允许。

示例：

- 生成运营日报
- 导出 SKU 建议表
- 整理测试记录
- 生成活动准备表
- 保存 AI 报告
- 回写本地日志
- 生成下一轮任务清单

### 3.2 中风险动作

必须人工确认后执行。

示例：

- 批量生成标题候选
- 批量生成主图文案候选
- 生成活动报名材料
- 生成价格测试建议
- 生成库存清货建议
- 回写 ERP 备注字段

### 3.3 高风险动作

MVP 阶段只生成建议，不自动执行。

示例：

- 自动上架
- 自动改价
- 自动修改主图
- 自动报名活动
- 自动增加投放预算
- 自动下架商品
- 自动清空库存
- 自动对接真实店铺后台执行操作

## 4. 人工确认流程

```text
AI 输出建议
↓
系统生成任务草案
↓
判断风险等级
↓
展示输入依据、风险提示、预期结果
↓
用户确认 / 拒绝 / 修改
↓
RPA 执行低风险任务
↓
保存确认记录与执行日志
```

## 5. 确认页面需要展示的信息

每个需要确认的任务都应该展示：

- 任务类型
- 目标商品
- AI 建议
- 数据依据
- RAG 召回依据
- 风险等级
- 可能影响
- 是否可回滚
- 执行结果预览
- 用户确认按钮

## 6. 确认记录数据结构

```json
{
  "approval_id": "APPROVAL001",
  "task_id": "TASK001",
  "user_id": "USER001",
  "target_product_id": "P001",
  "action_type": "generate_daily_report",
  "risk_level": "low",
  "ai_reason": "库存充足，当前适合继续自然流测试。",
  "risk_notes": ["不涉及真实店铺改价", "仅生成日报和建议表"],
  "approval_status": "approved",
  "approved_at": "2026-06-14T12:00:00Z"
}
```

## 7. AI 输出要求

涉及人工确认的输出必须包含：

- 建议动作
- 数据依据
- 风险提示
- 不确定性说明
- 是否需要人工确认
- 不建议自动执行的原因

示例：

```json
{
  "suggested_action": "生成活动报名准备表",
  "basis": ["库存 200", "售价 29", "成本 10", "活动价建议 19.9"],
  "risk_notes": ["活动价接近保本线，需确认物流成本和退款损耗"],
  "requires_human_approval": true,
  "auto_execution_allowed": false
}
```

## 8. 设计结论

Human-in-the-loop 是本项目的安全边界。

> 让 AI 和 RPA 进入电商运营流程，不等于让系统无约束地替用户操作店铺。真正可落地的自动化，需要把判断、确认、执行和追溯拆开。
