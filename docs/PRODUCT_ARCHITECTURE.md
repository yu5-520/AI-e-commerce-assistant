# PRODUCT_ARCHITECTURE

## 产品定位

AI ERP 企业级电商经营 SaaS 底座是一个任务驱动型 AI 经营系统。

当前 V11.16 Demo 目标：把 V11.15.1 后端经营接口兜底、V11.15.2 前端商品真实接口和店铺卡片产品化收口为统一版本基线，验证真实报表 / 接口数据进入系统后，能否稳定生成经营对象、标签、趋势信号、少量真正需要人处理的执行任务、详情报告、复核留痕，并支持反复测试时的一键清空运行态。

## 当前主导航

- 总览：经营同步结果、今日执行任务、风险事项、待复核事项、完成进度。
- 数据：Excel / CSV / JSON 上传、字段映射、数据源同步、数据版本、导入记录、演示运行态清空。
- 经营：商品档案、竞品信号、上新测试、流量趋势、店铺标签、店铺权重。
- 任务：执行队列、接收、提交、复核、完成。低风险结果不进入任务栏。
- 日志：任务完成记录、复核记录、复盘候选、审计线索。
- 账号：老板、总管、运营、角色权限、店铺归属。
- 系统：健康检查、运行态诊断、Repository 状态、PostgreSQL cutover、LLM 状态。

## 核心用户角色

- 老板：查看经营结果、店铺权重、风险进度、总管复核结果和组织效率。
- 总管：拆分执行任务、派发任务、复核运营提交、沉淀经验。
- 运营：只接收高风险 / 高时效 / 需要人处理的执行任务，低风险信号在商品或店铺档案中查看。

## 当前主业务链路

```text
数据导入 / 接口同步
→ 当前账号识别
→ 字段映射 / 校验
→ DataVersion / imported_report_rows / snapshots
→ 商品主档 operating_products
→ 店铺主档 operating_stores
→ 商品标签 / 店铺标签 / 店铺权重
→ 趋势和业务信号 business_signals_v6
→ 高风险高时效任务门控
→ 执行队列
→ 任务详情报告
→ 证据提交 / 总管复核
→ 日志留痕 / RAG 记忆候选
→ 导入闭环反查
```

## 前端产品规则

```text
总览不是数据库状态页，不展示“已入库多少条记录”等后端明细。
经营模块不是任务入口，商品 / 竞品 / 上新 / 流量进入各自对象页。
店铺前端只显示真实店铺名称，不显示工程 ID。
经营页店铺状态必须一店一行，横向分区展示店铺、标签、商品状态和执行任务。
商品档案必须通过 AppApi.product 读取后端清洗商品，不再默认读取 AppMockData.products。
商品档案展示清洗后的商品列表、商品标签和基线状态。
竞品信号是机会池，不直接生成任务压力。
上新测试管理主图、标题、SKU、卖点测试。
流量趋势先看曝光、点击、转化、ROI 和流量结构。
低风险信号沉淀为商品 / 店铺标签，不进入任务栏。
高风险高时效才进入前端执行队列。
接口异常显示错误态，不展示本地业务兜底。
正常空数据展示“暂无数据”。
```

## 数据规则

```text
报表是批次，商品才是分析主体。
分析阶段由商品入库ID和历史周期数决定，不由报表上传次数决定。
新商品只做建档和基线校验，不做环比、同比、连比、连续下滑任务。
低风险信号沉淀为商品标签 / 店铺标签 / 经营画像，不进入任务栏。
高风险高时效才进入前端执行队列。
商品任务优先级必须结合店铺权重判断。
任务进入执行队列后，详情页必须能打开；深度报告失败时返回基础兜底详情。
清空演示数据必须清空完整派生运行态，不能只清原始导入行。
```

## 架构边界

- 前端当前主入口：`web_demo/`。
- 后端当前主入口：`src/api/main.py`。
- 当前 Demo 主存储：SQLite runtime。
- 生产迁移目标：PostgreSQL + Alembic + Repository。
- LLM / Agent 只增强判断、标签、报告和任务草案，不直接执行真实经营动作。
- 任务状态、审批、复核、日志必须保持确定性。
- 低风险不是丢弃，而是写入标签和经营画像；前端任务栏只承接执行动作。
- 工程 ID 只用于后端匹配、权限、日志和审计，不在运营前端展示。
- `architecture.py` 中 V7/V8/V9 历史架构接口只作验收展示，不作为当前 Demo 主链路入口。

## 当前不做

- 不保留旧版本过程文档作为当前架构依据。
- 不把历史 V1-V9 作为当前主链路依据。
- 不让前端直接承载核心业务判断。
- 不让 LLM 直接执行投放、下架、改价等真实经营动作。
- 不在生产环境绕过 cutover check 直接切 PostgreSQL 主写。
