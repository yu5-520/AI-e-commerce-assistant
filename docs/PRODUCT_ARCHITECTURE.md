# PRODUCT_ARCHITECTURE

## 产品定位

AI ERP 企业级电商经营 SaaS 底座是一个任务驱动型 AI 经营系统。

系统以报表导入为触发器，以数据趋势和风险信号为判断依据，以任务池为主操作界面，以老板 / 总管 / 运营三端流转为执行链路，以日志、审计和 RAG 记忆作为复盘底座。

## 当前主导航

- 总览：今日任务台、风险事项、最新报表结果、待复核事项、完成进度。
- 数据：报表上传、字段映射、数据源同步、数据版本、导入记录。
- 经营：经营单元、商品、竞品、上新、流量、库存、售后。
- 任务：统一任务池、跨账号流转、接收、提交、复核、完成。
- 日志：任务完成记录、复核记录、复盘候选、审计线索。
- 账号：老板、总管、运营、角色权限、店铺归属。
- 系统：健康检查、Repository 状态、PostgreSQL cutover、LLM 状态。

## 核心用户角色

- 老板：查看经营结果、风险进度、总管复核结果和组织效率。
- 总管：拆分任务、派发任务、复核运营提交、沉淀经验。
- 运营：接收任务、处理任务、提交证据、补充数据。

## 主业务链路

```text
报表导入
→ 字段映射 / 校验
→ DataVersion
→ 模块投影
→ 趋势计算
→ 风险信号
→ Agent 经营档案 / 自动标签
→ 标签变化任务
→ 统一任务池
→ 跨账号流转
→ 证据提交 / 总管复核
→ 日志留痕 / RAG 记忆候选
```

## 架构边界

- 前端当前主入口：`web_demo/`。
- 后端当前主入口：`src/api/main.py`。
- 当前 Demo 主存储：SQLite runtime。
- 生产迁移目标：PostgreSQL + Alembic + Repository。
- LLM / Agent 只增强判断、标签、报告和任务草案，不直接执行真实经营动作。
- 任务状态、审批、复核、日志必须保持确定性。

## 当前不做

- 不保留旧版本过程文档。
- 不把历史 V1-V9 作为当前架构依据。
- 不让前端直接承载核心业务判断。
- 不让 LLM 直接执行投放、下架、改价等真实经营动作。
- 不在生产环境绕过 cutover check 直接切 PostgreSQL 主写。
